### Changed
- Reef database review comments - added mutation `db_review_comment` and read-only/batchable `db_review_comments` for append-only local notes on database objects and topics, including tables, columns, policies, triggers, publications, subscriptions, replication slots, and Supabase replication reviews. Comments are stored in the project SQLite database, never mutate the live DB, and are surfaced by `reef_scout` when relevant. Added migration `0032_project_db_review_comments` and smoke coverage in `test/smoke/reef-tooling.ts`.
- Reef database facts — added mutation `db_reef_refresh` to replace indexed Reef facts from the existing schema snapshot/read model, covering schemas, tables, columns, indexes, foreign keys, RLS policies, triggers, enums, RPCs, RPC-to-table refs, and indexed schema usages. Added `replaceReefFactsForSource` so source refreshes remove deleted DB facts instead of leaving stale rows, plus smoke coverage in `test/smoke/reef-tooling.ts`.
- Reef tooling follow-up — added mutation `diagnostic_refresh` to explicitly refresh selected diagnostic sources into Reef, mutation `finding_ack_batch` for reviewed acknowledgements in one round trip, and read-only/batchable `reef_overlay_diff` for comparing indexed vs working-tree facts. Added read-only/batchable `reef_instructions` for scoped AGENTS/.mako instruction lookup and `rule_pack_validate` for `.mako/rules` authoring feedback with descriptor previews. Claude Code metadata and focused smoke coverage (`test/smoke/reef-tooling.ts`) cover the new surfaces.
- Reef Engine Phases 7-10 — Model-facing Reef views. Added read-only, batchable `reef_scout`, `reef_inspect`, `project_open_loops`, `verification_state`, `project_conventions`, `rule_memory`, `evidence_confidence`, and `evidence_conflicts`. `context_packet` now consumes accepted/candidate convention facts as `reef_convention` candidates, adds deterministic `evidenceConfidenceLabel` metadata to working-tree/indexed/convention-backed candidates, and recommends open-loop, verification, and confidence follow-up tools. Claude Code metadata now includes the new Reef views, with `reef_scout` always-loaded. Smoke coverage in `test/smoke/reef-model-facing-views.ts` exercises Reef 7 through Reef 10, `tool_batch`, and context-packet convention consumption.
- Reef Engine Phase 6 — Performance boundary and native-engine decision. Added `test/smoke/reef-performance-boundary.ts`, a 5k indexed-file / 5k active-finding fixture that measures project finding queries, one-file finding queries, cold Reef-backed `context_packet`, and changed-file overlay replacement against the Reef thresholds. Current measured p95s are 58.82 ms, 0.54 ms, 240.38 ms, and 11.09 ms respectively, so the checked-in decision is to keep Reef TypeScript-only with no native package/install impact.
- Reef Engine Phase 5 — Tool view migration. `context_packet` now returns additive `activeFindings` from relevant active Reef findings, `ast_find_pattern` skips non-fresh indexed files through the Reef freshness guard to prevent stale phantom matches, `project_index_status` returns additive `reefFacts` for working-tree file snapshot fact freshness, and `git_precommit_check` keeps staged Reef finding persistence behind the shared migration rollback switch. `MAKO_REEF_BACKED=legacy` disables the migrated Reef enrichments for one release; `MAKO_REEF_BACKED=tool_a,tool_b` enables only selected migrated views.
- Reef Engine Phase 4 — Incremental watch engine. The existing index refresh coordinator now writes `working_tree_overlay` file snapshot facts for dirty paths before running path-scoped refresh, resolves active file-scoped indexed/working-tree findings for watcher-deleted files, and `ProjectIndexWatchState` reports the last overlay fact update/count/resolved-finding-count/duration/error plus the last refresh decision (`paths` vs `full`), fallback reason, and path counters. Smokes assert edits produce one replacement working-tree Reef fact, deletes produce a deleted snapshot fact plus resolved file findings, a fresh hot-index cache rebuilds from durable indexed facts, and changed-file overlay fact replacement reports under the current 500 ms Reef budget on the watch fixture.
- Reef Engine Phase 3 — Working tree and staged overlays. `git_precommit_check` now treats staged content as the source of truth with structured staged change records and deleted/renamed finding resolution. Added `working_tree_overlay`, an advisory mutation that persists live `working_tree` `file_snapshot` facts for changed files without a full reindex; added read-only, batchable `project_facts` and `file_facts`; and taught `context_packet` to consume existing working-tree overlay facts, label candidates with working-tree vs indexed overlay metadata, and recommend the overlay mutation when changed files lack live facts. Smokes cover staged overlay behavior, working-tree fact replacement/deletion, fact reads, batch access, and context-packet overlay fallback.
- Added `git_precommit_check`, a read-only staged-file guard for Git hooks and agents. It reads staged TS/TSX blobs, checks unprotected Next API routes and server/client boundary mistakes with TypeScript ASTs, merges discovered project-profile auth guards/server-only modules with `.mako/git-guard.json` or input overrides, and exposes `agentmako git precommit` for hook-friendly `continue` / `stopReason` output.

- Initial Testing Roadmap Phase 2 — MCP Perf: Project Store Lifetime. Eliminates the per-tool-call open-close cost that made `agentmako mcp` feel slower than comparable long-lived MCP servers on larger project DBs. Two structural changes. **(1)** `ProjectStore.close()` no longer runs `PRAGMA wal_checkpoint(TRUNCATE)` on every call — on a 78 MB WAL that was 100–300ms of synchronous bookkeeping per close. A new explicit `ProjectStore.checkpoint({ truncate?: boolean })` method replaces it: callers invoke it at shutdown (MCP stdio server SIGINT/SIGTERM/exit), not on every tool call. Default checkpoint mode is `PASSIVE` (non-blocking, does not shrink the WAL); `truncate: true` runs `TRUNCATE` mode for clean shutdown. SQLite's auto-checkpoint at 1000 pages continues to bound the WAL during normal operation. **(2)** New `ProjectStoreCache` in `packages/store/src/project-store-cache.ts` — per-process pool of open `ProjectStore` handles keyed by resolved project DB path. `borrow(options)` opens on first call for a given path and returns the same handle on subsequent calls; `flush()` checkpoints (with `truncate: true`) and closes every pooled store, idempotent. Cache is opt-in via a new `ToolServiceOptions.projectStoreCache` field plumbed through `withProjectContext` in `packages/tools/src/entity-resolver.ts`, `writeToolInvocationLogs` in `packages/tools/src/tool-invocation-logging.ts`, and the runtime-telemetry capture hook in `packages/tools/src/runtime-telemetry/capture.ts`. When a cache is present these paths borrow (no close on return); when absent, behavior is unchanged (open-close per call — the HTTP transport, CLI one-shots, and every existing smoke keep pre-Phase-2 semantics). `services/api/src/mcp-stdio.ts` creates a cache on startup, threads it into `ToolServiceOptions` for every tool invocation, and installs SIGINT / SIGTERM / transport-close / process-exit handlers that flush the cache (WAL checkpoint + close every pooled store). Smokes: `test/smoke/project-store-cache.ts` (borrow dedup, flush idempotency, post-flush reject, checkpoint safety, `close()` no-longer-TRUNCATE regression guard, logging cache regression guard), `test/smoke/mcp-perf-store-lifetime.ts` (seeds a 50-file project, times 10 warm `ast_find_pattern` calls in both modes, asserts cached mean is ≤85% of open-close mean OR within 1 ms — ratio-based to avoid CI host flakiness; local measurement showed cached at ~67% of open-close). Both wired into `test:smoke`. Existing smokes unaffected (they do not plumb a cache). HTTP transport caching is a clean follow-up if real workloads show it matters; the MCP stdio path is the hot path for Claude Code / Cursor / Codex. See `devdocs/roadmap/version-initial-testing/phases/phase-2-mcp-perf-store-lifetime.md`.
